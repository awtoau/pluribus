# Diamond Lattice ‚Äî batch flow for MachXO2 IOLOGIC fuzzing

## What this is

Diamond (Lattice's closed-source synthesis + PAR tool) is used here for two purposes:

1. **IOLOGIC fuzzing** ‚Äî synthesise designs using `IDDRXE` and other IOLOGIC
   primitives to extract their bitstream encoding.  prjcombine and yowasp do not
   support `IDDRXE` on MachXO2; Diamond is the only way to get routed bitstreams.

2. **H06E routing reports** ‚Äî Diamond's PAR report shows exactly which downstream
   CIB tiles tap the H06E horizontal bus for each ADC input pad net.  prjtrellis
   records only the source arc (JQ‚ÜíH06E); Diamond fills the gap.

## TCL API (diamondc)

Diamond's batch interpreter is `diamondc` (runs `pnmainc` under the hood).
Key TCL commands:

```tcl
prj_project open "path/to/project.ldf"   # load project
prj_run Synthesis -impl impl1             # synthesise only
prj_run MAP       -impl impl1             # map only
prj_run PAR       -impl impl1             # place and route (runs syn+map first)
prj_run Export    -impl impl1 -task Bitgen # generate bitstream
prj_project close
```

`prj_run PAR` runs all prior steps automatically (Synthesis ‚Üí MAP ‚Üí PAR).
`prj_run Export -task Bitgen` reads the PAR output and writes the `.bit` file.

## MachXO2 IOLOGIC primitives

The correct DDR input primitive for MachXO2 is **`IDDRXE`** (not `IDDRX1F` which
is ECP5).  Port mapping:

```verilog
IDDRXE u0 (
    .D    (pad_in),   // serial input from pad
    .SCLK (clk),      // sample clock
    .RST  (rst),      // sync reset ‚Äî must be a wire, not inline 1'b0
    .Q0   (q_rise),   // data captured on rising edge
    .Q1   (q_fall)    // data captured on falling edge
);
```

**Known gotcha**: Diamond LSE rejects `RST(1'b0)` inline ‚Äî assign a `wire rst = 1'b0`
instead.  The synthesiser creates multiple drivers otherwise.

In the bitstream, `IDDRXE` sets:
```
enum: IOLOGIC{B,C}.MODE IDDR_ODDR
enum: IOLOGIC{B,C}.CLKIMUX INV
enum: IOLOGIC{B,C}.CLKOMUX INV
enum: IOLOGIC{B,C}.GSR DISABLED
enum: IOLOGIC{B,C}.LSROMUX LSRMUX
```

This appears in the tile for the assigned pad (e.g. `PT12:PIC_T0` for pin 87/88).

## Extracting bitstream config (prjtrellis ecpunpack)

```bash
LD_LIBRARY_PATH=debris/tmp/prjtrellis/libtrellis/build \
  debris/tmp/prjtrellis/libtrellis/build/ecpunpack \
  --db debris/tmp/prjtrellis/database \
  fpga/diamond/impl1/aw2_impl1.bit \
  fpga/diamond/impl1/aw2_impl1_iddr.config
```

`run_diamond` does this automatically after a successful build.

## Adding new fuzz targets

1. Edit `iddr_fuzz.v` ‚Äî add more `IDDRXE` instances or try `ODDRXE`, `IDDRX2E`, etc.
2. Update `iddr_fuzz.lpf` ‚Äî add pin constraints for new ports.
3. Run `fpga/diamond/scripts/run_diamond`.
4. Diff the new `.config` against the baseline to see what bits changed.

## prjcombine contribution

The `IDDRXE` ‚Üí `IDDR_ODDR` mode encoding extracted here is the data needed to
contribute MachXO2 IOLOGIC support to prjcombine.  The contribution path:

1. Extract config bits for each IOLOGIC mode (`IDDRXE`, `ODDRXE`, `IDDRX2E`)
2. Map tile + bit position to prjcombine's MachXO2 device model
3. Submit PR to `prjcombine` with the new `iologic.rs` entries

Once prjcombine has this, nextpnr-machxo2 can generate the correct bitstream
for `IDDRXE` natively ‚Äî no Diamond needed.  Yosys synthesis is already fine;
it passes `IDDRXE` through as a black box.  The gap is purely in nextpnr's
bit-database (prjcombine), not in synthesis.
