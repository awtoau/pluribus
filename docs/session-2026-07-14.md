## 1. verilog.py — what was completed

### 1.1 Clock-domain FF naming

All 1105 FFs now emit clock-domain names instead of synthetic `reg_rNcN` grid
positions. The change is in `resolve_cell()`:

- **Before:** `reg_r3c13_2 <= 1'b0`
- **After:** `adc_wr_l__r3c13_A0 <= 1'b0`

Name format: `<clk_short>__<tile>_<slice>` where `clk_short` strips the
`clk_hN_` prefix (so `clk_h2_adc_wr_l` → `adc_wr_l`).

The `cell_clkname_map` was extended to cover all FFs (previously excluded cells
that already had a human name, but all 1105 FFs have synthetic `reg_rNcN`
human names so nothing was being populated). The regex `_SYNTHETIC_NAME_RE =
re.compile(r"^reg_r\d+c\d+")` gates the fall-through to the clock-domain path.

### 1.2 LUT truth table simplification

`_simplify_lut()` detects LUTs with ≤ 2 live inputs (non-NC, non-constant) and
emits direct Verilog operators:

```verilog
assign lut_xor_r9c3_Dk1 = reg_r3c4_q_7 ^ reg_r7c3_q_6;   // XOR
assign lut_n974_xor_n1350 = a & b;                          // AND
assign lut_inv = ~a;                                         // INV
```

26 XOR / 113 AND/NAND/NOR/etc assigns emitted; 486 localparams remain for
4-input logic that can't be simplified.

### 1.3 LUT localparam naming fix

`_lut_lut_lut_` triple-prefix bug fixed: the sanitised base name now strips
leading `lut_` repetitions before prepending `_lut_`, and human names
(non-synthetic) are preferred over cell ids for the base.

### 1.4 FF Q-wire simulation fix

FF register identifiers (`spi_rd__r5c1_A0`) are distinct from the Q output
wire names (`reg_r5c1_q_0` etc). Connect-assigns bridge them:

```verilog
assign reg_r3c13_q_2 = adc_wr_l__r3c13_A0;  // Q output
```

Skipped for: port names (3 FFs), dual-driven nets (15 FFs where a LUT also
assigns the same wire), and bus-shared Q nets (seen once per unique wire).

### 1.5 Current V07.v state

- 9517 lines, 0 iverilog errors, 0 warnings
- All 1105 FFs named by clock domain
- Output: `tmp/V07.v`

## 3. prjtrellis EBR.MODE local fix

### What was patched

The `EBR.MODE` config_enum in `bits.db` had a wrong bit address: all three
active modes were encoded as `F1B35` but Diamond 3.14 actually sets `F1B33 +
F1B34`. This caused `ecpunpack` to emit `unknown: F1B33` / `unknown: F1B34`
for every EBR block instead of `EBR.MODE PDPW8KC`.

**Fix applied (F1B35 → F1B33 F1B34) in:**
- `/home/dan/opt/share/trellis/database/MachXO2/tiledata/EBR1/bits.db`
- `/home/dan/opt/oss-cad-suite/share/trellis/database/MachXO2/tiledata/EBR1/bits.db`

Backups saved as `bits.db.orig` alongside each.

**NOT yet patched (inside awto-2000 — live RE project, cannot touch directly):**
- `/mnt/2tb/git/awto-2000/debris/tmp/prjtrellis/database/MachXO2/tiledata/EBR1/bits.db`

This is the copy that `load.py` actually uses (set by `TRELLIS_DBROOT` env var
in the load.py header comment). To apply the patch to the production copy:

```bash
cd /mnt/2tb/git/awto-2000
python3 -c "
import re, pathlib
p = pathlib.Path('debris/tmp/prjtrellis/database/MachXO2/tiledata/EBR1/bits.db')
t = p.read_text()
patched = re.sub(
    r'^((?:DP8KC|FIFO8KB|PDPW8KC)\s+F0B13(?:\s+F\dB\d+)*)\s+F1B35\s*$',
    r'\1 F1B33 F1B34', t, flags=re.MULTILINE)
p.write_text(patched)
print('done, replaced', t.count('F1B35') - patched.count('F1B35'), 'occurrences')
"
```

### EBR.MODE is more complex than documented

The `docs/unknown-bits-analysis.md` claimed all 7 V07 EBRs have identical
unknowns (`F0B13 F1B8 F1B20–22 F1B33 F1B34`). Checking the actual `.config`
shows 5 of 6 decoded EBR blocks have **additional** unknown bits involving
`F1B32`:

| EBR tile | Unknown bits in .config | After patch |
|----------|------------------------|-------------|
| EBR_R6C11 | F0B13+F1B8+F1B20-22+**F1B33+F1B34** | → PDPW8KC ✓ |
| EBR_R6C5  | F0B13+F1B8+F1B20-22+**F1B34** only | Still unknown |
| EBR_R6C15 | F0B13+F1B8+F1B20-22+**F1B32+F1B33+F1B34** | Still unknown |
| EBR_R6C2  | F0B13+F1B8+F1B20-22+**F1B32+F1B33** | Still unknown |
| EBR_R6C8  | F0B13+F1B8+F1B20-22+**F1B32+F1B34** | Still unknown |
| EBR_R6C21 | **F1B32 only** | Still unknown |

`F1B32` is a third mode bit the analysis doc missed — the 5 EBRs with it need
a proper re-fuzzing campaign to identify (different DATA_WIDTH or address-space
configurations probably set different combinations). Update `docs/unknown-bits-analysis.md`
with this finding.

### EBR.MODE does not affect arc stitching

The missing DAC D0/D2/D6/D7 nets (`pad_36/38/42/43`) are **not** caused by the
EBR.MODE decode issue. `load.py` stitches EBR read arcs by pattern-matching
JC/JD port names — it does not gate on `EBR.MODE`. The 140 EBR port arcs and
30 EBR read nets in the V07 database are present and correctly stitched
regardless.

## 4. Missing DAC net root cause (definitive)

Investigation via the V07 database (`tmp/test-run.db`) found:

**DAC pins 36/38/42/43 (DAC_D0/D2/D6/D7):**
- `pad_map.net_in = None` — IO buffer fabric-side input not resolved
- `net_fanout` shows: `n2501 → PAD pad_36 via JA2` — the fabric net IS
  connected to the IO buffer input
- `n2501` (= DAC_D0) has **no upstream driver** in `net_fanout`, `ffs`, or
  `luts` — nothing outputs `n2501` in the recovered netlist

**Trace through arcs:**
```
arcs for n2501:
  R11C10  V02N0201 → H00R0000  (both = n2501)
  R11C10  H00R0000 → JA2       (both = n2501)
```

The routing chain ends at `V02N0201 → H00R0000 → JA2` (IO buffer input). The
arc that should feed V02N0201 from the fabric is not decoded.

**Root cause: CIB bottom-edge routing gap (frames F24–F27).**

These four pads sit in the bottom IO row (R11). The CIB tiles adjacent to
bottom-edge pads have routing configuration in frames F24–F27 that prjtrellis
does not model (never fuzzed with high-speed IO standards). The arcs connecting
the internal fabric to the IO buffer input exist in silicon but are invisible
to `ecpunpack`.

**This is NOT fixable by patching bits.db.** It requires a fuzzing campaign:
vary `BASE_TYPE`/`DRIVE` on bottom-edge pads and observe which F24–F27 routing
bits change. Document in `docs/unknown-bits-analysis.md` as an additional
category alongside the existing CIB_PIC_B_DUMMY section.
